## Character Layer

This page exists because the distinction matters before the full doctrine is
written.

The repo has a real guide for [Behavior Design](./behavior-design.md). This
page is for the next layer up: the part that turns a clear behavioral contract
into a more distinctive interface.

## Why This Topic Has A Shelf

Teams eventually run into a second design problem after they get behavior
mostly right: how an agent should feel to work with.

That is where character starts to matter. Not as fluff. As a force that can
make an agent easier to correct, easier to trust, or much more annoying than it
needs to be.

## What A Finished Version Should Answer

- how personality differs from behavior design
- when character improves usability versus when it becomes noise
- how to make an agent memorable without making it less correctable
- how to keep style from overwhelming process, review, and evidence

## Current Boundary

For now, the rule is simple:

- behavior design is in scope now
- character layer is real, but still early
- process still outranks style

This page is here so the topic has a proper home instead of sneaking into every
other doc as an unresolved vibe argument.
